How Schema Drives Validation
validate_vault.py binds all constants directly from _schema at module-load time. validate_file() uses these bindings for all checks. If schema changes, a process restart is required — no live reload.

How Schema Drives Template Generation
generate_templates.py calls its own load_schema(vault) (not core.shared). It reads VALID_TYPES, looks up TYPE_TO_CONSTANT[note_type] → constant name → reads that constant from schema module. Calls build_type_section_map(), then render_template() → writes files.

How Schema Drives Task Generation
upgrade_vault.py uses SECTION_MAP (for Family B), TRACKED_SECTIONS (indirectly via has_* field checks), VALID_DIFFICULTIES, DOMAIN_PRIORITY_WEIGHT, SUBDOMAIN_DIFFICULTY (via getattr fallbacks). The schema's VALID_DIFFICULTIES content controls which difficulty branch executes (the "advanced" scoring path is dead code in the demo vault since VALID_DIFFICULTIES = frozenset({"intermediate"})).

Idempotency Mechanism
String equality check: if existing == content: unchanged. The content is deterministic (sorted types, fixed template format), so re-runs produce identical bytes given identical schema.

Overwrite vs Merge
Full overwrite — no merging. The existing file's content beyond headings is discarded. Templates contain only # {{Title}} and ## Heading lines with blank lines between them.

Post-generation Validation
Reads each written file back and compares extracted headings against section_map[note_type]. Mismatches cause sys.exit(1). This is a write-then-verify pattern.

7. Task Generation System
Responsible module: core/shared/upgrade_vault.py
Entry function: generate_tasks(records: list[dict]) -> list[dict]

Trigger Conditions
A record generates tasks if and only if: rec.get("status") == "partial". Complete notes are skipped entirely.

For core-concept notes:
check_how_it_works(body): calls extract_section_body(body, "## How It Works"). If None → "missing-how-it-works". If present but < 3 lines matching ^\d+\.\s+ → "weak-how-it-works".
check_tradeoffs(body): calls extract_section_body(body, "## Trade-offs"). If None → "missing-tradeoffs". If present but < 3 data rows (table lines minus 2) OR header cells ≠ ["aspect","benefit","cost"] → "weak-tradeoffs".
check_key_principles(fields): checks fields.get("has_key_principles") is not True → "missing-key-principles". This checks the YAML boolean field, not the actual section body.
For Family B notes:
check_family_b_sections(body, note_type): gets SECTION_MAP[note_type], compares to set(find_headings(body)). Returns list of missing headings.
Priority Calculation (core-concept):

base = 0.0if "advanced" in VALID_DIFFICULTIES and difficulty == "advanced": base += 3.0elif difficulty == "intermediate": base += 1.0for issue:  if tradeoffs:     base += 3.0  if how_it_works:  base += 2.0  if key_principles: base += 1.0score = base * DOMAIN_PRIORITY_WEIGHT.get(domain, 1.0)
Note: In the demo vault, "advanced" not in VALID_DIFFICULTIES, so the if "advanced" in... branch never executes. All notes score +1.0 for difficulty at most.

Sorting

tasks.sort(key=lambda t: (-t["score"], t["path"]))
Primary: score descending. Secondary: path ascending (alphabetical, case-sensitive string sort).

Reliance on Filesystem Ordering
discover_files() uses os.walk() (OS-ordered) then explicitly sorts filenames per directory and finally sorts the full list. These two sort orders can diverge for mixed-case paths.

Claim 1: No Randomness
VERDICT: VALID

No random, uuid, secrets, or os.urandom calls anywhere in the codebase
No time.time() or datetime.now() in output-affecting code paths
time.monotonic() used only for metrics/timeouts, not in data output
Claim 2: Repeatable Outputs
VERDICT: PARTIAL

Proven repeatable:

discover_files(): os.walk output is sorted per-directory, then globally. Given fixed filesystem state, identical output.
generate_tasks(): sort is (-score, path). Score is a deterministic float function of field values. Path is a string. Output is repeatable for fixed input.
Template generation: content is "\n".join(["# {{Title}}", ""] + [section, "" for section in sections]) + "\n" — deterministic string concatenation.
Partial concern:

os.walk() ordering is OS/filesystem-dependent before the explicit sort. The content_files.sort() call sorts Path objects using Python's default Path.__lt__, which is case-sensitive on all platforms. The MCP index sorts by path.lower() (case-insensitive). On a case-insensitive filesystem (Windows), two files A.md and a.md cannot coexist, but the sort order difference is a theoretical risk on Linux vaults.
The DOMAIN_PRIORITY_WEIGHT and SUBDOMAIN_DIFFICULTY attributes are fetched with getattr(_schema, 'X', {}). If absent from schema, silently defaults to {}. This is a hidden default — the behaviour changes silently if the schema adds these attributes.
Claim 3: Idempotent Template Generation
VERDICT: VALID

Template content is deterministic from schema
Idempotency check is existing == content (byte-level string equality)
The write uses filepath.write_text(..., newline="\n") — consistent line endings
Cleanup removes non-canonical files, so re-runs on a mutated template directory converge to correct state
Hidden State Risk
sys.modules["vault_schema"] persists for the process lifetime. If two CLI commands are run in the same Python process (hypothetically), the second would reuse the first's schema. In practice, run.py is a subprocess launcher via raise SystemExit, so this is not an active risk for CLI use.

Concurrency Risk
_adapter_lock serialises all adapter calls, preventing race conditions on sys.modules and os.chdir. This is correct but means adapter endpoints do not parallelize.
_get_vault_lock() provides per-vault locking for index operations using double-checked locking. The global _global_lock guards the _locks dict creation. This is correct.
_metrics_lock guards _request_count, _endpoint_counts, _response_times. _request_count is a module-level int modified with global statement inside a lock — correct.
_shutting_down flag is written without a lock (in signal handler and startup). This is a benign race for a boolean flag, but technically not atomic in Python before 3.12 GIL removal scenarios.

14. Performance Characteristics
discover_files() — O(N) where N = number of files
os.walk traverses the entire directory tree. Two sorts occur: sorted(filenames) per directory (O(k log k) per dir), then content_files.sort() globally (O(N log N)). For typical vault sizes (hundreds of notes), this is fast. No caching between calls — every command re-traverses the filesystem.

validate_file() — O(L) where L = lines in file
Per-file: 1 read, 1 frontmatter parse, multiple regex operations on body, section extraction (linear scan), find_headings (linear scan). No cross-file state. Total validation: O(N × L).

generate_tasks() — O(N × L + N log N)
load_all does N reads and parses. check_how_it_works/check_tradeoffs do section body extraction (linear). Final sort is O(N log N) on task count.

MCP build_index() — O(N × L)
Scans all files, stores {path, fields, body} for each. Body is stored in memory — this is the largest memory cost. The _estimate_index_size() function tracks growth via sys.getsizeof() + len(body). For 1000 notes at 5KB each, the index holds ~5MB of body text in memory.

MCP get_index() — O(1) cached, O(N×L) on rebuild
Schema hash check: reads schema file bytes → SHA-256. This is O(schema_size) per call when outside the cooldown window (every 2+ seconds).

MCP query() — O(N + K log K) where K = match count
Linear scan of index, with a 200ms timeout guard. After filtering, sort by path (O(K log K)), then slice for pagination. The timeout means large vaults under heavy filter load may return partial results.

Adapter endpoints — O(N×L) always
Adapters never use the in-memory index. Every call to /validation, /tasks, /notes, /summary, /gaps performs a full filesystem scan from scratch, plus the sys.modules manipulation overhead. These are serialised (single-threaded via _adapter_lock).

Scalability
Vaults with thousands of notes: the CLI tools scale linearly and are acceptable for batch operations
MCP primary endpoints (/query, /note): scale well due to in-memory index
MCP adapter endpoints: do not scale — each call is O(N) filesystem reads, serialised

16. Design Risks & Hidden Coupling
Risk 1: CWD-Dependent Schema Loading
Location: core/shared/__init__.py:load_schema()
Risk: The entire CLI layer depends on os.chdir(scripts_dir) being called before any core/shared/*.py module is imported. If any import occurs before the chdir (e.g., if a module is added to COMMANDS and has a top-level import that triggers core.shared), load_schema() will search from the wrong CWD, fail to find vault_schema.py, and raise FileNotFoundError.

Coupling: run.py → os.chdir → load_schema() → vault_schema.py. Removing or reordering the chdir breaks all CLI commands silently (they raise FileNotFoundError instead of printing a useful message).

Risk 2: sys.modules["vault_schema"] Global Cache
Location: core/shared/__init__.py:load_schema()
Risk: The schema is cached under the plain key "vault_schema" in sys.modules. If two different vault schemas are loaded in the same process (not currently possible via CLI, but possible if the CLI is extended), the second load returns the first vault's schema. The MCP adapter correctly clears this with sys.modules.pop("vault_schema", None) before each adapter call.

Evidence: validation_adapter.py line: sys.modules.pop("vault_schema", None) confirms this is a known risk, not theoretical.

Risk 3: sys.modules Manipulation in _adapter_call
Location: mcp/server/mcp_server.py:_adapter_call()
Risk: This function removes all core.* modules from sys.modules, calls the adapter, then restores them. If the adapter raises an exception after partial sys.modules mutation and before the finally block executes, the finally block still runs (correct). However, if the adapter leaves partially-initialised modules in sys.modules (e.g., a module that failed mid-execution), the restoration logic removes any new core.* keys not in saved, but does not check for partially-executed modules.

Coupling: This creates tight coupling between the MCP server's module cache state and the adapter's behaviour. Adding a new adapter that imports additional core.* submodules could silently break module isolation.

Risk 4: os.chdir in Adapter Calls
Location: mcp/core/adapters/*.py:_prepare_env()
Risk: os.chdir(scripts_dir) changes the process-wide CWD. This is protected by _adapter_lock, but it means no other code that depends on CWD (e.g., relative file operations) can run concurrently. The _adapter_call wrapper saves and restores CWD in finally, but the CWD change is still process-global for the duration of the adapter call.

Risk 5: TYPE_TO_CONSTANT Hardcoded in generate_templates.py
Location: core/generate_templates.py, lines defining TYPE_TO_CONSTANT
Risk: The mapping from note type to schema constant name is hardcoded in the template engine, not in the schema. If a new note type is added to vault_schema.py that is not in TYPE_TO_CONSTANT, the template engine hard-fails with sys.exit(1). This creates a maintenance dependency: every new note type requires a corresponding entry in generate_templates.py.

Risk 6: getattr(_schema, 'DOMAIN_PRIORITY_WEIGHT', {}) Silent Defaults
Location: core/shared/analyse_vault.py, core/shared/upgrade_vault.py, core/shared/discover_missing.py
Risk: Multiple schema attributes are fetched with getattr(..., {}) fallback. If these attributes are added to one vault's schema but not another, the system behaves differently across vaults without any error. The default {} means all domain weights default to 1.0 (via .get(dom, 1.0)), which may not be the intended behaviour for a vault that defines partial weights.

Risk 7: Subprocess in Contract Checks
Location: mcp/core/system_contract.py:check_vault_audit, check_vault_validate, check_vault_inject_dry
Risk: Three contract checks spawn subprocesses with subprocess.run([sys.executable, str(script_file)]). The inject_frontmatter.py script, when run without --dry-run, would modify vault files. The contract check always passes --dry-run. If the argument order or the script's argument parsing changes, the contract check could trigger live writes. The scripts use "--dry-run" in sys.argv (not argparse), which is robust to positional changes.

Risk 8: CORE_CONCEPT_FIELDS_NO_SUBDOMAIN Defined but Not Used
Location: vault_schema.py
Risk: CORE_CONCEPT_FIELDS_NO_SUBDOMAIN is defined in the schema but is not referenced anywhere in validate_vault.py. The validation logic uses CORE_CONCEPT_FIELDS (which includes subdomain) and handles subdomain absence via a depth check (if field_name == "subdomain" and depth < 3: continue). The constant is dead code.

Risk 9: Family B scoring uses SECTION_MAP presence check
Location: upgrade_vault.py:check_family_b_sections()


expected = SECTION_MAP.get(note_type, ())if not expected:    return []
Risk: If a Family B type has an empty sections tuple in SECTION_MAP, it is silently treated as "no sections to check" and generates no tasks. This is indistinguishable from a missing SECTION_MAP entry.
